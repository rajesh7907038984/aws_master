2. MIXED ASYNC/SYNC PATTERNS IN SCORM API
   Severity: CRITICAL
   Impact: Race Conditions, Data Inconsistency, Promise Resolution Errors
   
   Issue:
   - SCORM API calls use synchronous XMLHttpRequest (correct for SCORM 1.2)
   - Save & Exit operations use asynchronous fetch() API
   - This creates race conditions where content calls finish before save completes
   
   Evidence:
   - File: scorm/templates/scorm/player_storyline.html (lines 164, 205, 444)
   - File: scorm/templates/scorm/player_rise360.html (lines 171, 236, 509)
   - File: scorm/templates/scorm/player_captivate.html (lines 164, 222, 442)
   - File: scorm/templates/scorm/player_generic.html (lines 164, 199, 440)
   
   Code Pattern:
   ```javascript
   // SCORM API - Synchronous (CORRECT)
   const xhr = new XMLHttpRequest();
   xhr.open('POST', API_ENDPOINT, false);  // false = synchronous
   
   // Save & Exit - Asynchronous (PROBLEMATIC)
   fetch('{% url "scorm:sync_on_exit" %}', {
       method: 'POST',
       ...
   }).then(response => {
       // May execute AFTER SCORM content has already called LMSFinish
   });
   ```
   
   Root Cause:
   - SCORM content expects immediate synchronous response from API
   - Async fetch() doesn't block, allowing content to proceed
   - Content may call LMSFinish() before sync completes
   - Results in lost suspend_data, scores, and bookmarks
   
   Fix Required:
   - Convert all sync operations to synchronous XMLHttpRequest
   - OR use navigator.sendBeacon() for guaranteed delivery on page unload
   - Add loading state to prevent multiple simultaneous saves
   - Implement save queue to handle rapid save requests

5. SCORE SYNC SERVICE STRICT FILTERING CAUSING DATA LOSS
   Severity: CRITICAL
   Impact: Valid Scores Not Being Saved, Users Losing Progress
   
   Issue:
   - Score sync service has VERY strict filtering in _should_sync_score()
   - Only syncs when lesson_status is 'completed' or 'passed'
   - Valid partial scores (e.g., 60% on first attempt) are ignored
   - Users must complete 100% to get ANY score saved
   
   Evidence:
   - File: scorm/score_sync_service.py, lines 164-205
   - Code comment: "DO NOT sync just because user viewed the content"
   - Multiple checks prevent sync unless status='completed' or 'passed'
   
   Code Pattern:
   ```python
   def _should_sync_score(attempt: 'ScormAttempt') -> bool:
       # Only sync if actually completed or passed (NOT failed without score)
       if attempt.lesson_status in ['completed', 'passed']:
           return True
       
       # For failed status, only sync if there's an actual score
       if attempt.lesson_status == 'failed' and attempt.score_raw is not None and attempt.score_raw > 0:
           return True
       
       # ❌ DO NOT sync just because user accessed content
       return False
   ```
   
   Problem:
   - Many SCORM packages set lesson_status='incomplete' during assessment
   - User takes quiz, gets 60%, but status is still 'incomplete'
   - Sync is skipped because status != 'completed' or 'passed'
   - Score lost when user navigates away
   
   Fix Required:
   - Modify _should_sync_score() to sync ANY valid score >= 0
   - Add sync for 'incomplete' status if score exists
   - Sync progress_percentage > 0 even without completion
   - Keep strict filtering only for zero-score empty attempts


6. CMI DATA EXTRACTION FAILURES IN EXIT SYNC
   Severity: HIGH
   Impact: Bookmarks Lost, Quiz State Not Saved, Resume Fails
   
   Issue:
   - sync_on_exit() attempts to extract data from CMI when fields are None
   - No validation that CMI data exists before accessing
   - String operations on None values cause TypeErrors
   - Silent failures prevent any data from being saved
   
   Evidence:
   - File: scorm/auto_sync_views.py, lines 176-212
   - Code attempts: str(cmi_score).strip() when cmi_score is None
   - Code attempts: len(attempt.suspend_data) when suspend_data is None
   
   Code Pattern:
   ```python
   # Extract score from CMI if not in model
   if not attempt.score_raw:
       score_key = 'cmi.core.score.raw' if version == '1.2' else 'cmi.score.raw'
       cmi_score = attempt.cmi_data.get(score_key)
       if cmi_score and str(cmi_score).strip():  # ⚠️ FAILS if cmi_score is not a string
           try:
               from decimal import Decimal
               attempt.score_raw = Decimal(str(cmi_score))
   ```
   
   Fix Required:
   - Add type checking before string operations
   - Use safe_str() helper function for all conversions
   - Validate CMI data structure before extraction
   - Log extraction failures without blocking save


7. CACHE INVALIDATION INCONSISTENCIES
   Severity: HIGH
   Impact: Stale Data Displayed, Gradebook Shows Wrong Scores
   
   Issue:
   - Multiple cache key patterns used inconsistently
   - Cache cleared in some places but not others
   - Cache invalidation uses pattern matching which may miss keys
   - Race conditions when cache cleared during read operation
   
   Evidence:
   - File: scorm/score_sync_service.py, lines 139-146
   - File: scorm/auto_sync_views.py, lines 274-286
   - File: courses/views.py, line 5415 (cache_key pattern)
   
   Inconsistent Patterns:
   ```python
   # Pattern 1: score_sync_service.py
   cache_keys = [
       f'gradebook_course_{topic.courses.first().id}_*',
       f'topic_progress_{topic.id}_*',
       f'user_progress_{scorm_attempt.user.id}_*'
   ]
   
   # Pattern 2: auto_sync_views.py
   cache.delete(f'gradebook_course_{ct.course.id}')
   cache.delete(f'topic_progress_{topic.id}_{request.user.id}')
   
   # Pattern 3: views.py
   cache_key = f"course_progress_{course_id}_{request.user.id}"
   ```
   
   Fix Required:
   - Standardize cache key naming across all modules
   - Create centralized cache management class
   - Use versioned cache keys to force invalidation
   - Implement cache stampede protection


8. JSON FIELD NULL VALUE HANDLING
   Severity: HIGH
   Impact: Database Errors, Save Operations Fail
   
   Issue:
   - JSON fields (completed_slides, navigation_history, etc.) can be None
   - Code assumes these are always lists/dicts
   - Leads to TypeErrors when trying to access/modify
   - Django JSONField doesn't guarantee empty collections vs None
   
   Evidence:
   - File: scorm/auto_sync_views.py, lines 215-224
   - Code sets fields to [] or {} only in sync_on_exit, not consistently
   
   Code Pattern:
   ```python
   # CRITICAL: Ensure all JSON fields have valid values (never None)
   if attempt.completed_slides is None:
       attempt.completed_slides = []
   if attempt.navigation_history is None:
       attempt.navigation_history = []
   ```
   
   Problem:
   - These checks only in sync_on_exit(), not in other save paths
   - API handler may save with None values
   - Other code assumes these are never None
   
   Fix Required:
   - Set default=list/dict in model field definitions
   - Add database-level constraints to prevent NULL
   - Create migration to fix existing None values
   - Add model clean() method to enforce non-None
