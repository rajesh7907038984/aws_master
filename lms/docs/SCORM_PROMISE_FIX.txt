SCORM [object Promise] URL Bug Fix
===================================
Date: October 12, 2025

ISSUE IDENTIFIED:
-----------------
SCORM content URLs were being generated with "[object Promise]" instead of actual file paths:
- Wrong: https://staging.nexsy.io/scorm/content/97/scormcontent/[object%20Promise]
- Correct: https://staging.nexsy.io/scorm/content/97/scormcontent/index.html

ROOT CAUSE:
-----------
The SCORM API injection in scorm/views.py (lines 385-488) was using ASYNC functions:
- All API methods (Initialize, GetValue, SetValue, etc.) were defined as "async function"
- Legacy SCORM content (like Articulate Storyline's scormdriver.js) calls these methods SYNCHRONOUSLY
- When calling an async function without "await", it returns a Promise object
- JavaScript concatenates Promise as string → "[object Promise]"

Example from scormdriver/indexAPI.html:
```javascript
var bookmark = GetBookmark();  // Expected: "index.html", Got: Promise
window.location.href = "../scormcontent/" + bookmark;  // Results in: "../scormcontent/[object Promise]"
```

THE FIX:
--------
1. Converted all SCORM API methods from ASYNC to SYNCHRONOUS (scorm/views.py)
   - Changed from: async function + await fetch()
   - Changed to: regular function + synchronous XMLHttpRequest
   - xhr.open('POST', endpoint, false) - The "false" parameter makes it synchronous

2. Added _getCookie helper function for CSRF token retrieval

3. Improved fix_scorm_relative_paths() function logic:
   - Better handling of multiple path replacements
   - Added regex-based replacement to avoid double-fixing paths
   - Prevents creating duplicate path prefixes

WHY SYNCHRONOUS XHR:
--------------------
While synchronous XHR is generally discouraged in modern web development:
1. SCORM specification requires immediate return values
2. Legacy SCORM content cannot be modified to use async/await
3. SCORM API calls are small, fast operations
4. This is the standard approach for SCORM LMS implementations
